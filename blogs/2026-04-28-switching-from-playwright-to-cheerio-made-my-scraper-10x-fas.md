The website was UGAP (Union des Groupements d&apos;Achats Publics), France&#x2019;s <a href="https://www.ugap.fr/" rel="noreferrer">public procurement marketplace</a>. So I built an Actor - like I usually do - that does the following:</p><ul><li>Searches for a list of keywords (for example, table)</li><li>Collects product links from search results</li><li>Visits product pages and extracts title, prices (HT/TTC - pre-tax and tax-included), description, and reference code</li><li>Pushes clean structured data into an Apify dataset</li></ul><p>As usual, Playwright was my first choice, as it handles almost any site. But that flexibility comes at a cost: it&apos;s slow, consumes a lot of memory, and the Docker image is large.</p><p>This January, instead of getting fit myself, I reshaped this Actor to use <code>CheerioCrawler</code> instead of Playwright. That single decision changed everything: 10x faster, 10x lower memory usage, better stability, lighter builds, and much higher concurrency on the Apify platform.</p><p>This article is my migration story, based on the actual code.</p><h2 id="why-ugap-was-a-good-target">Why UGAP was a good target</h2><p>UGAP is a central purchasing body for French public procurement. It has a huge catalog (hundreds of thousands of items) and the data is available in static HTML, which makes it a good target for scraping.</p><p>General rule here: public-sector websites usually serve their data in static HTML since their goal is to let people access the data easily (and for anyone) and not to make it difficult to scrape.</p><p>No need to log in, no need to interact with private areas.</p><figure class="kg-card kg-image-card kg-card-hascaption"><a href="https://console.apify.com/actors/MhDQxIodrxH6LmlcI?addFromActorId=MhDQxIodrxH6LmlcI"><img alt="Switching from Playwright to Cheerio made my scraper 10x faster" class="kg-image" height="243" src="https://storage.ghost.io/c/f2/6e/f26ec999-9a90-4aee-a0d4-9b3ca2bb668f/content/images/2026/04/data-src-image-f536f8de-df6d-4c6b-967a-c4453f1598ef.png" width="2000" /></a><figcaption><span style="white-space: pre-wrap;">UGAP Scraper on Apify Store</span></figcaption></figure><h2 id="prerequisites">Prerequisites</h2><p>This isn&apos;t a beginner&apos;s guide to web scraping. I&#x2019;m assuming you&#x2019;re comfortable with Node.js and you&#x2019;ve used Apify or <a href="https://crawlee.dev/"><u>Crawlee</u></a> before. What you need:</p><ul><li>Node.js 18+</li><li>Basic familiarity with Crawlee routers and the Apify SDK</li></ul><p>Useful docs I reference in this article:</p><ul><li><a href="https://crawlee.dev/js/docs/deployment/apify-platform" rel="noreferrer">Crawlee on the Apify platform guide </a></li><li><a href="https://crawlee.dev/api/cheerio-crawler/class/CheerioCrawler" rel="noreferrer">CheerioCrawler docs</a></li><li><a href="https://docs.apify.com/api/v2" rel="noreferrer">Apify API v2</a></li></ul><p>I&apos;m not going to mention proxy settings, dataset handling or any logging in this article. The only focus is on the performance.</p><h2 id="where-i-started-playwrightcrawler">Where I started: PlaywrightCrawler</h2><p>When I created the project, I used the Playwright-based Actor template. To be honest, I hadn&apos;t yet looked closely at how UGAP loaded product details - I just defaulted to what I knew. My reasoning at the time was simple: I wanted robustness. If the site rendered prices via JavaScript, a browser would be the easiest path. I also wanted a single scraping approach for both search and product pages.</p><p>So my initial stack looked like this:</p><ul><li><code>PlaywrightCrawler</code> in <code>src/main.ts</code></li><li><code>createPlaywrightRouter()</code> in <code>src/routes.ts</code></li><li>Extractors built around Playwright&#x2019;s Page (so everything was async and browser-driven)</li></ul><p>It worked. I could run a small input and get correct results. But performance could be better.</p><h3 id="the-performance-problem-with-playwright">The performance problem with Playwright</h3><p>With a browser, each &#x201c;request&#x201d; is a navigation inside a real Chromium instance. Even if you technically increase concurrency, you&#x2019;re fighting CPU usage, RAM usage, and execution time. </p><p>And per-request latency is dominated by overhead. For every page, Playwright needs to create or reuse a browser context, navigate to the page, wait for the load state (I was using <code>networkidle</code> in the product handler), and evaluate selectors inside the browser context. </p><p>All of this added up, and the Actor was slow.<strong> </strong>Here&apos;s what the migration looked like in numbers.</p><p>Before (with Playwright): </p><figure class="kg-card kg-image-card"><img alt="Switching from Playwright to Cheerio made my scraper 10x faster" class="kg-image" height="140" src="https://storage.ghost.io/c/f2/6e/f26ec999-9a90-4aee-a0d4-9b3ca2bb668f/content/images/2026/04/data-src-image-d3918a31-33fa-4d01-96e9-82ccb8962f1e.png" width="1086" /></figure><p>After (with Cheerio): </p><figure class="kg-card kg-image-card"><img alt="Switching from Playwright to Cheerio made my scraper 10x faster" class="kg-image" height="150" src="https://storage.ghost.io/c/f2/6e/f26ec999-9a90-4aee-a0d4-9b3ca2bb668f/content/images/2026/04/data-src-image-f0aa0fd6-077a-4dd6-b033-1277f94959f7.png" width="1098" /></figure><p>The execution time dropped from <strong>3 minutes to 14 seconds</strong>. As you can see, the memory usage is considerably lower with Cheerio.</p><h2 id="dry-january-static-is-a-good-fit-for-cheerio">Dry January: static is a good fit for Cheerio</h2><p>As I mentioned earlier, the static nature of the website made CheerioCrawler a good alternative to test. Sure, it would require me to rewrite some parts of the Actor, but it would be worth it. I treated it as an experiment.</p><p>If you&#x2019;ve never migrated from a browser crawler to Cheerio, here&#x2019;s the core mental shift: with Playwright, you <strong>&#x201c;drive&#x201d; a browser</strong> like a user; with Cheerio, you just <strong>parse HTML</strong> as text.</p><h2 id="the-migration-what-i-changed-in-practice">The migration: what I changed in practice</h2><p>I touched very few files:</p><ul><li><code>package.json</code> (and <code>package-lock.json</code>)</li><li><code>src/main.ts</code></li><li><code>src/routes.ts</code></li><li><code>src/utils/extractors.util.ts</code> (where the logic for extracting data from pages is)</li></ul><h3 id="1-dependencies-removing-playwright-from-the-runtime">1) Dependencies: removing Playwright from the runtime</h3><p>Before the migration, the Actor depended on:</p><ul><li><code>@crawlee/playwright</code></li><li><code>playwright</code></li><li>A <code>postinstall</code> that installed Playwright browsers - a heavy install that slowed down every build.</li></ul><p>After the migration, the only runtime dependency was <code>@crawlee/cheerio</code>.</p><p>This is the part people underestimate. Removing Playwright isn&#x2019;t just about speed - it changes package size, installation time, and build time. And the main advantage is that you can now iterate faster on your Actor.</p><h3 id="2-maints-switch-playwrightcrawler-%E2%86%92-cheeriocrawler">2) <code>main.ts</code>: switch <code>PlaywrightCrawler</code> &#x2192; <code>CheerioCrawler</code></h3><p>With Playwright, my main entry was essentially:</p><ul><li>Create proxy config</li><li>New <code>PlaywrightCrawler({ requestHandler: router, ... })</code></li><li>Run the start URLs</li></ul><p>The Cheerio version has the same structure, just with different options.</p><p>Before:</p><pre><code class="language-ts">const crawler = new PlaywrightCrawler({
    proxyConfiguration,
    maxRequestsPerCrawl,
    requestHandler: router,
    launchContext: {
        launchOptions: {
            args: [
                &apos;--disable-gpu&apos;, 
            ],
        },
    },
});
</code></pre><p>After:</p><pre><code class="language-ts">const crawler = new CheerioCrawler({
    proxyConfiguration,
    maxRequestsPerCrawl,
    requestHandler: router,
    minConcurrency: 10,
    maxConcurrency: 50,
    requestHandlerTimeoutSecs: 30,
    maxRequestRetries: 3,
});
</code></pre><h3 id="3-routests-switch-page-%E2%86%92">3) <code>routes.ts</code>: switch <code>page</code> &#x2192; <code>$</code></h3><p>The handler signature changes a little bit. So there was some code to update.</p><h4 id="search-handler-browser-evaluation-%E2%86%92-html-link-parsing">Search handler: browser evaluation &#x2192; HTML link parsing</h4><p>In Playwright, I used:</p><ul><li><code>page.waitForSelector(&apos;a[href*=&quot;-p&quot;]&apos;)</code></li><li><code>page.$$eval(&apos;a[href*=&quot;-p&quot;]&apos;, ...)</code> to get product links</li></ul><p>Before:</p><pre><code class="language-ts">router.addHandler(&apos;search&apos;, async ({ request, page, enqueueLinks, log }) =&gt; {
    await page.waitForSelector(&apos;a[href*=&quot;-p&quot;]&apos;, { timeout: 10000 })
    .catch(() =&gt; {
        log.warning(&apos;No product links found on search results page&apos;, { url: request.loadedUrl });
    });
});
</code></pre><p>In Cheerio, no need to wait for the page to load. I switched to:</p><pre><code class="language-ts">router.addHandler(&apos;search&apos;, async ({ request, $, enqueueLinks }) =&gt; {
    const productLinks: string[] = [];
    $(&apos;a[href*=&quot;-p&quot;]&apos;).each((_, element) =&gt; {
        const href = $(element).attr(&apos;href&apos;);
        const absoluteUrl = new URL(href, request.loadedUrl).href;
        productLinks.push(absoluteUrl);
    });
});</code></pre><h4 id="extractorsutilts-async-calls-%E2%86%92-synchronous-calls"><code>extractors.util.ts</code>: async calls &#x2192; synchronous calls</h4><p>In the Playwright version, my extractors looked like:</p><ul><li><code>extractTitle(page): Promise&lt;string&gt;</code></li><li><code>extractPrices(page): Promise&lt;{ priceHT, priceTTC }&gt;</code></li></ul><p>After migrating, the extractors became synchronous:</p><ul><li><code>extractTitle($): string</code></li><li><code>extractPrices($): { priceHT, priceTTC }</code></li></ul><p>Same logic as applied in the <code>routes.ts</code> file.</p><h2 id="why-the-speedup-hit-10x">Why the speedup hit ~10x</h2><p>By speedup, I mean <strong>crawl throughput</strong>: how many pages (search + product) I can process per unit time on the same budget. That throughput is a product of two things:</p><ol><li><strong>Per-request latency</strong></li><li><strong>Safe concurrency</strong></li></ol><p>On static sites, Playwright moves both in the wrong direction.</p><h3 id="per-request-latency-browser-tax-vs-raw-html-fetch">Per-request latency: browser tax vs. raw HTML fetch</h3><p>On a static page:</p><ul><li>An HTTP fetch + HTML parse can take tens of milliseconds to a few hundred milliseconds.</li><li>A Playwright navigation often lands in the seconds range once you include load state, network settling, and page lifecycle overhead.</li></ul><p>Even if the server is fast, the browser workflow is doing more work than needed.</p><h3 id="concurrency-50-parallel-http-requests-vs-a-few-pages-at-once">Concurrency: 50 parallel HTTP requests vs. a few pages at once</h3><p>With <code>CheerioCrawler</code>, running <code>maxConcurrency: 50</code> is normal. With <code>PlaywrightCrawler</code>, 50 concurrent pages is usually a stress test, not a default. You can do it, but you&#x2019;ll pay in memory, CPU, and instability.</p><p>So the &#x201c;10x&#x201d; story is not one knob. It&#x2019;s the combination:</p><ul><li>Lower latency per request</li><li>Much higher safe concurrency</li><li>Lower memory footprint, which keeps autoscaling happy</li><li>Fewer timeouts and retries, which reduces wasted work</li></ul><h2 id="what-surprised-me">What surprised me</h2><p>A few things stood out.</p><h3 id="the-migration-was-small">The migration was small</h3><p>Switching from Playwright to Cheerio touched fewer than 200 lines of code.</p><figure class="kg-card kg-image-card"><img alt="Switching from Playwright to Cheerio made my scraper 10x faster" class="kg-image" height="413" src="https://storage.ghost.io/c/f2/6e/f26ec999-9a90-4aee-a0d4-9b3ca2bb668f/content/images/2026/04/data-src-image-a63d8258-4935-424e-a05c-123b72ca7bec.png" width="2000" /></figure><p>See the <a href="https://github.com/guillim/apify-actor-ugap/commit/2d03c6f6640b25bc3a8183d10eddeffaeed96986" rel="noreferrer">git diff here</a>:&#xa0;</p><ul><li>Switch the crawler from <code>PlaywrightCrawler</code> to <code>CheerioCrawler</code></li><li>Switch the router from <code>page</code> to <code>$</code></li><li>Switch the extractors from async to synchronous</li></ul><h3 id="synchronous-router-is-much-simpler">Synchronous router is much simpler</h3><p>Playwright routers often include waiting logic:</p><ul><li>Wait for the selector</li><li>Wait for the load state</li><li>Handle partial renders</li></ul><p>Once I moved to Cheerio, those concerns disappeared. The page is either in the HTML, or it isn&#x2019;t.</p><h3 id="the-build-got-a-lot-lighter">The build got a lot lighter</h3><p>Removing <code>postinstall</code> browser setup is underrated. It reduced friction immediately: The final image is around 450MB, down from about 2.3GB.</p><h2 id="what-i%E2%80%99d-do-differently-next-time">What I&#x2019;d do differently next time</h2><p>Two things I&apos;d change on the next Actor:</p><ul><li>I would validate &#x201c;static vs. dynamic&#x201d; on day zero</li><li>I would design for easier &#x201c;crawler swap&#x201d;, keeping separation of concerns in mind</li></ul><h2 id="conclusion">Conclusion</h2><p>I didn&#x2019;t switch from Playwright to Cheerio because Playwright is bad. I switched because <strong>it was the wrong tool for this specific job</strong>.</p><p>UGAP serves the product data I need in static HTML. That fact makes <code>CheerioCrawler</code> the natural fit, as it removes the browser overhead, unlocks real concurrency, and cuts memory dramatically. </p><p>If you&#x2019;re building Actors and you want a free performance upgrade, here&#x2019;s my recommendation:</p><ul><li>Start with website inspection: static or dynamic?</li><li>Use the lightest crawler that can do the job.</li><li>Keep your extraction logic modular so you can switch later.</li></ul><p>That&#x2019;s how my Actor is now 10x faster, thinner, and less RAM hungry: all thanks to picking the right tool.</p><h2 id="faq">FAQ</h2><h3 id="should-i-always-replace-playwrightcrawler-with-cheeriocrawler">Should I always replace <code>PlaywrightCrawler</code> with <code>CheerioCrawler</code>?</h3><p>No.&#xa0;Only do it when your target pages are server-rendered and the fields you need exist in the raw HTML.</p><h3 id="how-do-i-know-if-the-content-is-in-static-html">How do I know if the content is in static HTML?</h3><p>Use view source or fetch the page HTML directly (curl, HTTP client). If the fields you need are missing and only appear after JS runs, you likely need Playwright.</p><p>You can also test it by disabling JavaScript in your browser and checking if the fields you need are still present. In Chrome, you can do this in the settings as shown below:</p><figure class="kg-card kg-image-card kg-card-hascaption"><img alt="Switching from Playwright to Cheerio made my scraper 10x faster" class="kg-image" height="1248" src="https://storage.ghost.io/c/f2/6e/f26ec999-9a90-4aee-a0d4-9b3ca2bb668f/content/images/2026/04/data-src-image-921b29c9-89ca-4df2-b318-64d482ee524f.png" width="1338" /><figcaption><span style="white-space: pre-wrap;">Test JavaScript in Chrome</span></figcaption></figure><h3 id="can-i-mix-both-approaches">Can I mix both approaches?</h3><p>Yes. A hybrid approach can be the best of both worlds:</p><ul><li><code>CheerioCrawler</code> for most pages</li><li><code>PlaywrightCrawler</code> for a small subset that truly requires interaction or JS rendering</li></ul><p>Your Docker build image will be considerably bigger, but it can be worth it if you need to scrape a lot of pages that are static and some that are dynamic.</p><h3 id="what%E2%80%99s-the-biggest-win-from-switching">What&#x2019;s the biggest win from switching?</h3><ul><li>Lower RAM</li><li>Higher concurrency</li><li>Fewer timeouts</li><li>Lighter builds</li></ul><h3 id="is-the-ugap-actor-available-publicly">Is the UGAP Actor available publicly?</h3><p>Yes, you can find it on Apify Store: <a href="https://apify.com/anchor/apify-actor-ugap">UGAP Website Scraper</a></p><h3 id="what-do-the-results-look-like">What do the results look like?</h3><p>Here&apos;s a preview of the dataset with extracted title, prices, reference code, and image:</p><figure class="kg-card kg-image-card kg-card-hascaption"><img alt="Switching from Playwright to Cheerio made my scraper 10x faster" class="kg-image" height="787" src="https://storage.ghost.io/c/f2/6e/f26ec999-9a90-4aee-a0d4-9b3ca2bb668f/content/images/2026/04/data-src-image-84b44437-5978-40a4-ae82-51a1676a11bc.png" width="2000" /><figcaption><span style="white-space: pre-wrap;">Dataset preview showing extracted title, prices, reference, and image</span></figcaption></figure>
